Steel Thread Recipe

How we extracted a service from a 200k-line .NET codebase using AI-SDLC methodology. Copy this pattern for your own project.

The Challenge

Codebase
nopCommerce — 200k+ line open-source .NET e-commerce platform
Problem
ShoppingCartService.cs — a "god class" with ~2,000 lines and 31 constructor dependencies
Goal
Extract add-to-cart operations into a clean ICartService without breaking anything
Method
AI-SDLC Steel Thread extraction (vertical slice through the full stack)

The 5 Steps (The Recipe)

1
🎯 Identify the Steel Thread
Pick the narrowest user flow that touches every architectural layer. For nopCommerce: the "add item to cart" flow (UI → Controller → Service → Repository → Database).
Artifact: A clear scope statement — what's in, what's out.
2
🔍 Trace with LSP, Not Grep
Use Language Server Protocol queries (find-references, document-symbols, type-hierarchy) to map every caller and dependency. AI used LSP to find all 21 callers across Nop.Web, Nop.Services, Nop.Web.Framework, and 5 plugins.
Artifact: research.md (~2k lines compressed from 200k)
3
📐 Plan the Extraction
Agent reads research.md (and ONLY research.md) to produce a 15-step implementation plan. Each step has verification commands. The plan correctly identified: new ICartService interface, clean DI registration, unidirectional dependency flow.
Artifact: plan.md with ordered steps + risk register
4
⚡ Implement with Harnesses Running
Agent follows plan.md step-by-step. After every change: C# compiler runs (catches type errors), unit tests run (catches logic errors), Playwright E2E runs (catches behaviour changes). Agent self-corrects from each failure.
Artifact: Working code across 21+ files, all tests green
5
🛡️ Review the "Physics Beat Law" Moments
The design doc said to keep GetProductsRequiringProductAsync in the old service. But that would have created a circular dependency. The C# compiler (physics) forced the AI to override the design doc (law) and move the method. This is exactly how it should work.
Artifact: Updated design docs reflecting reality

What Made It Work

Three Key Factors

  • Dense handoffs — application-design.md and code-generation-plan.md constrained the AI across 21 file changes without hallucination
  • Fresh contexts per phase — Research, Plan, and Implement each got a new context with only the previous phase's artefact
  • Physics over law — The compiler caught what the design doc missed. Three layers of automated verification (compiler + unit tests + E2E) caught three different classes of error.

Try It Yourself

1
Pick Your Steel Thread
Choose the narrowest flow through your system — the one that touches UI, service layer, data access, and database.
2
Run the Research Phase
Use LSP to trace every file, class, and method involved: @workspace Trace the [X] flow from UI to database.
3
Feed to Plan & Implement
Feed research.md to a fresh Plan phase context, then implement with your test suite running continuously.
